# Music Gradebook

Серверное Django-приложение для музыкальной школы с русскоязычным интерфейсом. Система покрывает учебный процесс: роли пользователей, предметы и зачисления, журнал и оценки, домашние задания, полугодовые планы, видео-библиотеку, а также отправку ученических ответов по заданиям.

## Что умеет проект

- Авторизация на Django.
- Роли через профиль пользователя: `ADMIN`, `TEACHER`, `STUDENT`, `PARENT`.
- Курсы и привязка преподавателей/учеников.
- Журнал: задания, оценки, комментарии, просмотр результатов.
- Домашние задания:
  - создание заданий преподавателем;
  - композиции/произведения в домашнем задании;
  - перенос активных произведений из предыдущего домашнего задания;
  - выбор произведений из полугодового плана;
  - защита от дублей при переносе и импорте.
- Полугодовые планы и цели.
- Видео-библиотека.
- Ученические ответы на задания:
  - текстовый комментарий;
  - загрузка видео;
  - отображение ответа у преподавателя при проверке;
  - сохранение видео в общей Library.
- Простая speech-to-text интеграция на базе Vosk для основных текстовых полей.

## Текущая логика по домашним заданиям

### Перенос произведений между домашними заданиями

Если преподаватель добавил произведения в домашнее задание ученика, то при создании следующего домашнего задания для того же ученика и того же курса активные произведения подхватываются автоматически.

### Импорт из полугодового плана

При создании домашнего задания преподаватель может выбрать:

- `I полугодие`
- `II полугодие`

После этого система показывает произведения/цели из соответствующего полугодового плана, и преподаватель может добавить нужные позиции в текущее домашнее задание. Если название уже есть среди перенесённых произведений, дубль не создаётся.

### Ответ ученика по заданию

Ученик может открыть задание и отправить:

- текстовый ответ;
- видео.

После отправки:

- ответ сохраняется у назначения задания;
- статус задания обновляется;
- преподаватель видит ответ в своей части проверки;
- видео попадает в Library как видео-ответ на домашнее задание.

## Speech-to-Text (Vosk)

В проекте реализована минимальная русская speech-to-text интеграция:

- возле основных текстовых полей показывается кнопка микрофона;
- браузер записывает WAV;
- аудио отправляется на backend;
- backend распознаёт речь через Vosk;
- распознанный текст вставляется обратно в поле.

### Где используется

- формы создания домашнего задания;
- поля описания и названия;
- строки добавления произведений;
- формы полугодовых планов;
- текстовый ответ ученика там, где это подключено в шаблоне.

## Установка Vosk-модели

### Что нужно установить

Python-зависимость уже описана в `requirements.txt`:

```bash
pip install -r requirements.txt
```

### Какую модель использовать

Нужна русская модель Vosk. Для этого проекта рекомендуется использовать только компактную модель:

- `vosk-model-small-ru-0.22`
- Ссылка на скачивание: `https://alphacephei.com/vosk/models/vosk-model-small-ru-0.22.zip`

Важно: внутри папки модели должны лежать реальные файлы модели, например:

- `am/final.mdl`
- `conf/model.conf`
- `graph/words.txt`

### Куда класть модель

Рекомендуемое место в репозитории:

```text
models/
  vosk/
    vosk-model-small-ru-0.22/
      am/
      conf/
      graph/
      ivector/
      ...
```

Важно, чтобы не было лишней вложенности после распаковки архива. Неправильно:

```text
models/vosk/vosk-model-small-ru-0.22/vosk-model-small-ru-0.22/am/...
```

Правильно:

```text
models/vosk/vosk-model-small-ru-0.22/am/...
```

### Как проект ищет модель

Код speech-to-text пытается найти модель в таком порядке:

1. Путь из `VOSK_MODEL_PATH`
2. Родительская директория `VOSK_MODEL_PATH`
3. Стандартная папка проекта `models/vosk`

Если в `models/vosk` лежит одна валидная Vosk-модель, проект сможет её подобрать автоматически.

### Переменная окружения для сервера

Для production/staging лучше явно задать путь к модели:

Linux:

```bash
export VOSK_MODEL_PATH=/opt/music-gradebook/models/vosk/vosk-model-small-ru-0.22
```

Windows:

```powershell
$env:VOSK_MODEL_PATH='C:\path\to\music-gradebook\models\vosk\vosk-model-small-ru-0.22'
```

### Важная особенность Windows

На Windows Vosk может падать, если путь к модели содержит не-ASCII символы, например кириллицу в пути вроде `Рабочий стол`.

Для этого в проекте уже добавлен обходной механизм:

- приложение находит модель в репозитории;
- если путь содержит не-ASCII символы, модель копируется во временную ASCII-безопасную папку;
- Vosk загружает модель уже оттуда.

Пример временного кэша на Windows:

```text
C:\Users\<user>\AppData\Local\Temp\music-gradebook-vosk\<имя_папки_модели>
```

Это значит:

- в репозитории модель можно хранить как обычно;
- на Windows не нужно вручную переносить её в другую папку;
- при обновлении модели можно очистить временный кэш, если нужно принудительно пересоздать копию.

### Если модель обновили

На Windows при необходимости можно удалить временный кэш:

```text
C:\Users\<user>\AppData\Local\Temp\music-gradebook-vosk
```

После следующего запроса распознавания система создаст его заново.

## Что нужно для сервера

Минимальный чек-лист для DevOps:

1. Развернуть код проекта.
2. Создать и активировать виртуальное окружение.
3. Выполнить:

```bash
pip install -r requirements.txt
python manage.py migrate
python manage.py collectstatic --noinput
```

4. Создать директории для:
   - `media/`
   - `staticfiles/`
   - Vosk-модели
5. Положить русскую Vosk-модель в `models/vosk/...` или указать точный путь через `VOSK_MODEL_PATH`.
6. Убедиться, что процесс приложения имеет права на чтение модели и запись в системный temp-каталог.
7. Прописать корректные:
   - `DJANGO_SECRET_KEY`
   - `DJANGO_ALLOWED_HOSTS`
   - `DJANGO_CSRF_TRUSTED_ORIGINS`
   - `DJANGO_DEBUG=False`
8. Перезапустить сервис приложения после деплоя.

## Миграции

После последних изменений обязательно выполнить:

```bash
python manage.py migrate
```

Причина:

- добавлена связь Library-видео с ученической отправкой по заданию.

## Что важно не сломать при деплое

- Ролевой доступ: преподаватель / ученик / родитель / админ.
- Media-файлы, потому что ученические видео и Library используют файловое хранилище Django.
- Доступ к Vosk-модели по файловой системе.
- CSRF и `ALLOWED_HOSTS` для рабочего домена.
